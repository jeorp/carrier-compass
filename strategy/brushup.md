# 記事ブラッシュアップガイド

## ペルソナ整合ブラッシュアップワークフロー

ペルソナ（`writers/[author].md`）を更新した後、または記事が一定数溜まった時点で実施する。

1. `writers/[author].md` を読む（最新ペルソナを確認）
2. `strategy/articles.md` で対象著者の全記事を一覧する
3. 各記事を読み、以下の観点でペルソナと照合する
   - 経歴・時系列の矛盾がないか（年齢・在籍期間・転職タイミングなど）
   - 口癖・価値観・行動原則と齟齬がないか
   - ペルソナ更新によって「抜け落ちた push/pull 要因」がないか（動機の説明が片面だけになっていないか）
4. 修正不要の記事は明示的に「問題なし」と記録する（沈黙は曖昧さになる）
5. 修正箇所は最小限のピンポイント編集。リライトしない
6. 記事から新たなペルソナ情報が生まれた場合は `writers/[author].md` も更新する

## 記事品質チェック

### 自動チェック（review.mjs でカバー済み）

`npm run review -- --all` で確認できる項目。

- `targetKeyword` の記事間重複
- `targetKeyword` がタイトル・description・本文冒頭100字に含まれているか
- description の文字数（120〜160字）
- `strategy/keywords.md` のテーブルと `author` フィールドの整合
- ですます混在（yuki は対象外）

### 手動チェック（読まないと判断できない項目）

**ペルソナ整合**
- [ ] 登場する経歴・数字がペルソナと一致しているか
- [ ] 一人称・口調がペルソナの文体と合っているか
- [ ] 感情の出方（照れて引く・力まず自然体など）がペルソナらしいか

**具体性**
- [ ] エピソードに固有の状況があるか（「失敗した」ではなく「入社4ヶ月目に〇〇して」レベル）
- [ ] 断言しない部分が意識的に残されているか（「正直まだわからない」など）
- [ ] 感情の揺れが入っているか（論理だけで進んでいないか）

**AI検出リスク**
- [ ] 文末が単調でないか（「〜ます」統一になっていないか）
- [ ] 論理接続詞（「つまり」「逆に言えば」「その分」）を多用していないか
- [ ] 3点セットの列挙（A・B・C）が多くないか
- [ ] H2・H3の見出し構造が整いすぎていないか（全H2に同数のH3はAIらしい）
- [ ] 末尾が読者への問いかけ（「あなたはどうですか？」）で終わっていないか

**重複表現チェック（著者の記事が5本溜まるたびに実施）**

下記の要注意フレーズを `grep` で全記事に対して検索し、2本以上にヒットしたら修正する。

```bash
# syun 要注意フレーズ
grep -r "背中がすっと冷えた" src/content/blog/
grep -r "3年後も同じ" src/content/blog/
grep -r "完璧なパスを待つな" src/content/blog/
grep -r "正しいポジションに立て" src/content/blog/
grep -r "誰かの人生が少し良くなった" src/content/blog/
grep -r "今週、誰かの役に立てたか" src/content/blog/
```

各フレーズの居場所（1本に絞る基準）：
- 「背中がすっと冷えた気がした。サウナなのに。」→ `career-change-trigger`
- 「このまま3年後も同じ」→ `career-change-trigger`
- 「完璧なパスを待つな」→ `fear-of-changing-jobs`
- 「正しいポジションに立て」→ `gave-up-high-salary`
- 「誰かの人生が少し良くなった」→ 複数可（ペルソナの核心なので3本まで許容）
- 「今週、誰かの役に立てたか」→ `gave-up-high-salary`・`career-change-trigger` の2本まで

**文章構造（マナブスタイル）**
syun・kz の記事に適用。詳細は [`strategy/writing-style-manabu.md`](writing-style-manabu.md) を参照。
- [ ] H2冒頭1文で結論を言い切っているか
- [ ] 感情語より事実・数字が先に来ているか
- [ ] 「良かった点から書く」のようなメタコメントがないか

**構成**
- [ ] 段落の長さにばらつきがあるか（1文段落と複数文段落が混在しているか）
- [ ] 締めが筆者自身の言葉・行動・独り言のような余韻になっているか

**記事構造のパターン化チェック（著者の記事が10本溜まるたびに実施）**

H2数の均一化：
- [ ] H2が4本の記事が7割を超えていないか。超えていたら一部の記事を3本または5本に調整する

見出し表現の重複：
- [ ] 「## 僕自身の〜」「## 自分の話をすると」が3本以上にないか
- [ ] 「〜の正体」「〜に共通していること」「〜の共通パターン」「〜はあるのか」が重なりすぎていないか

冒頭パターンの重複：
- [ ] 「〜という相談は多い。」で始まる記事が3本以上にないか
- [ ] 「〜、という話を聞く。」「〜、という人に会ったことがある。」系で始まる記事が固まっていないか

```bash
# 構造パターン重複チェック（syun）
grep -r "## 僕自身の" src/content/blog/
grep -r "## 自分の話をすると" src/content/blog/
grep -rn "^という相談は多い" src/content/blog/
grep -rn "^[^#].*という相談" src/content/blog/
```

修正方針：
- 「## 僕自身の〜」はその記事固有の状況を反映した見出しに変える（例：「## コンサルを辞めた最初の夜のこと」）
- 冒頭が「相談は多い」系なら、具体的なエピソードや問いかけから入る形に変える
- H2数の調整は、独立した短いセクションを前後のセクションに吸収するか、長いセクションを2つに割る

## 実施タイミングの目安

| トリガー | 対象 |
|---|---|
| ペルソナファイルを更新したとき | 該当著者の全記事 |
| 著者の記事が5本溜まったとき | 直近5本 |
| 記事の品質に違和感を覚えたとき | 該当記事のみ |
