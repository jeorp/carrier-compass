# carrier-note

## 概要
自分のキャリアに迷いや不安を抱える人に向けたブログ「キャリアの羅針盤」。完璧なルートではなく、修正しながら目的地へ向かうというコンセプトのもと、現役のキャリアコンサルタント・営業職・エンジニアのライター陣がリアルな現場視点で発信する。

## 技術スタック
- フレームワーク: Astro（静的サイトジェネレーター）
- 言語: TypeScript
- 記事管理: Markdown（`.md`）
- ホスティング: GCP Cloud Storage + Cloud CDN
- 記事生成補助: Claude API（Anthropic）※内部利用のみ

## セットアップ

```bash
# 依存関係のインストール
npm install

# 開発サーバー起動
npm run dev

# 静的ファイルのビルド
npm run build

# ビルド結果のプレビュー
npm run preview
```

## よく使うコマンド

```bash
# GCPへのデプロイ（distディレクトリをCloud Storageにアップロード）
gsutil -m rsync -r dist/ gs://<BUCKET_NAME>
```

## ライティングルール（AI検出回避）

記事を書く際は以下のルールを必ず守ること。

### 文章構造
- 段落の長さを意図的にバラつかせる。1文で終わる段落と、少し長い段落を混在させる
- 「まとめ」セクションを箇条書きにしない。締めは筆者の言葉で段落として書く
- H2・H3の見出し構造が整いすぎないようにする。すべての見出しに同じ数のH3が入るのはAIらしく見える

### 語彙・文末
- 「つまり」「逆に言えば」「その分」などの論理接続詞を多用しない
- 文末を「〜ます」に統一しない。「〜だった」「〜かもしれない」「〜なんですよね」など変化をつける
- 3点セットの列挙（A・B・C）を避ける。どうしても使う場合は2点か4点にする

### エピソード・具体性
- 体験談には必ず固有の状況を添える。「失敗した」ではなく「入社4ヶ月目に〇〇して怒られた」レベルの具体性
- 断言しない部分を意識的に残す。「正直まだわからない」「これは私の場合の話ですが」など
- 感情の揺れを入れる。「悔しかった」「なんか悲しくなった」など論理ではない感情の言葉

### 記事の締め方
- 末尾を「あなたはどう思いますか？」などの読者への問いかけで終わらせない。AIっぽく見える
- 筆者自身の言葉・行動・小さな独り言のような余韻で締める

### ペルソナごとの注意
- **yuki**: 完璧にまとまった文章にしない。途中で話が少し脱線したり、言い直したりする箇所を入れる。「なんか」「うまく言えないけど」など考えながら話している感じを自然に混ぜる。深刻になりすぎず、ゆるい空気感を保つ。ただし芯が見える瞬間を1〜2箇所作る。「でもこれだけは譲らない」「ここは変えなかった」という決断や価値観がさらりと出るタイミングを意識する。絵文字を自然に散りばめる（女性読者向け）。ただし句点「。」の直後に絵文字を置かない。文末に使う場合は句点なし、文頭に使う場合は先に絵文字を置く
- **syun**: 熱くなりすぎる手前でいったん照れて引く瞬間を作る

## ターゲットUX

**主要ターゲット：iPhone 16ユーザーが電車・ベッドで片手操作**

- デバイス：iPhone 16（393×852px）
- 状況：電車・ベッドでの片手（右手親指）操作
- 親指が楽に届く範囲：画面下部40〜50%（y=425px以下）
- 親指が届きにくい範囲：画面上部30%（y=255px以上）

この前提のもと、頻繁に使うインタラクション（フィルター・ナビ・リンク）は画面下部に置く。上部にあっていいのは一度だけ押す要素（ヒーローCTAなど）のみ。

## 記事管理

**新記事を企画・執筆する前に必ず以下の順で参照・更新すること。**

### ブラッシュアップワークフロー

ペルソナ更新後・著者記事が5本溜まった時点・品質に違和感を覚えたときに実施する。
詳細手順とチェックリストは [`strategy/brushup.md`](strategy/brushup.md) を参照。

### 新記事執筆ワークフロー

1. `strategy/keywords.md` からメインキーワードを選ぶ（未執筆テーマ一覧から優先度・競合度を確認）
2. `strategy/articles.md` で既存記事との重複がないか確認する
3. frontmatterに `targetKeyword` フィールドを追加する（例: `targetKeyword: "営業 辞めたい 続ける"`）
4. `npm run review -- --all` を実行し、`❌ [SEO: targetKeyword重複の可能性]` が出ないことを確認する。重複が出た場合は下記「targetKeyword重複への対応」を参照する
5. タイトル・description・本文冒頭100字に `targetKeyword` を自然に含める
6. 執筆後、`strategy/articles.md` と `strategy/keywords.md` の両方を更新する

### targetKeyword重複への対応

`npm run review -- --all` で重複が検出されたら、記事を消す前に以下を試みる。

1. **ハブ記事を1本決める**（重複グループの中で最も包括的・汎用的なテーマを扱う記事）
2. **スポーク記事のrelatedにハブを追加する**（スポーク → ハブへの一方向リンク）
3. **ハブ記事のrelatedにスポーク記事を追加する**（相互リンク完成）
4. それでも意図・内容が完全に被る場合はtargetKeywordを変更して差別化する

棲み分けの目安：「同じキーワードで競合する2本は、**体験談 vs 解説**・**男性視点 vs 女性視点**・**転職前 vs 転職後**のように切り口が違えば相互リンクで共存できる」
